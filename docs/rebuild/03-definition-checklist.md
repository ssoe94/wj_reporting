# 추가 정의 체크리스트

## 1. 목적

재구축에서 가장 흔한 실패 원인은 "구현은 시작했는데, 정작 기준이 없어서 다시 흔들리는 것"이다.

이 문서는 개발을 시작하기 전에 반드시 정의하거나, 최소한 임시 결정을 내려야 하는 항목을 정리한다.

## 2. 꼭 먼저 정의할 것

## 2.1 화면 인벤토리

모든 화면을 아래 항목으로 정리한다.

- 화면명
- 현재 URL
- 담당 부서
- 주요 사용자
- 화면 목적
- 읽기 전용인지 입력 화면인지
- 반드시 필요한지, 개선 대상인지, 제거 후보인지

최소 분류 예시:

- 분석
- 생산계획
- 사출
- 가공
- 품질
- 재고/영업
- ECO
- 관리자
- 현장단말

## 2.2 사용자/권한 표

현재 시스템에는 staff, admin 성격, 모듈별 view/edit 권한, 현장 단말 사용자 성격이 같이 있다.
재구축 전 아래 표가 필요하다.

- 사용자 유형
- 접근 가능한 메뉴
- 조회 가능 데이터
- 수정 가능 데이터
- 승인 가능 작업
- 다운로드 가능 여부
- AI 사용 가능 여부

권장 역할 예시:

- `super_admin`
- `module_admin`
- `production_manager`
- `quality_manager`
- `operator`
- `viewer`
- `field_terminal`

## 2.3 용어집

같은 개념을 여러 이름으로 부르는 부분을 먼저 고정해야 한다.

특히 아래 항목은 반드시 정의가 필요하다.

- `part_no`
- `model`
- `model_code`
- `material_code`
- `material_name`
- `specification`
- `line_no`
- `machine_no`
- `machine_name`
- `business_date`
- `report_dt`
- `plan_date`
- `snapshot_date`

각 용어마다 아래를 적는다.

- 업무 의미
- 저장 위치
- 입력 주체
- 표기 형식
- 화면 표기명

## 2.4 기준 시간 규칙

다음 질문에 답이 필요하다.

- 운영 기준 시간대는 무엇인가
- 생산일은 자정 기준인가, 교대 기준인가
- 오전 8시 이전 데이터는 어느 날짜에 귀속되는가
- 화면 표시 시간과 저장 시간은 어떻게 다른가
- 이메일 보고서 기준 시간은 무엇인가

이 항목은 재고, MES, 생산, AI 요약 정확도에 직접 영향을 준다.

## 2.5 공통 품목 마스터 기준

현재는 공통 마스터로 발전 가능한 구조가 있으므로, 여기를 중심축으로 둘지 먼저 결정해야 한다.

질문:

- 전사 공통 품목 기준은 `UnifiedPartSpec`로 갈 것인가
- 사출 전용/가공 전용/품질 전용 필드는 어떻게 분리할 것인가
- source system 우선순위는 무엇인가
- 비활성 품목은 어떻게 처리할 것인가

## 3. API 설계 전에 정의할 것

## 3.1 엔드포인트 목록표

각 API에 대해 아래를 정리한다.

- path
- method
- 도메인
- 설명
- 입력/조회/집계 구분
- 인증 필요 여부
- 권한
- 현재 사용 화면
- 차기 상태

차기 상태 예시:

- 유지
- 이름만 정리
- 응답 스키마 정리
- 신규 API로 대체
- 제거 후보

## 3.2 응답 필드 표준

적어도 아래는 공통 표준이 필요하다.

- id
- created_at
- updated_at
- display_name
- status
- is_active
- note

또한 enum 값도 문서로 고정하는 것이 좋다.

- 품질 `section`
- 품질 `judgement`
- 생산 `plan_type`
- 실행 `status`
- 가공 `supply_type`

## 3.3 에러 처리 규칙

정의할 것:

- validation error 형식
- unauthorized / forbidden 차이
- not found 처리
- business rule 위반 메시지
- 사용자 노출용 메시지와 로그용 메시지 분리

## 4. DB 정리 전에 정의할 것

## 4.1 테이블 분류

테이블을 아래처럼 나눈다.

- master
- transaction
- snapshot
- integration
- analytics helper

예시:

- `UnifiedPartSpec`: master
- `InjectionReport`, `AssemblyReport`, `QualityReport`: transaction
- `DailyInventorySnapshot`, `FinishedGoodsTransactionSnapshot`: snapshot
- MES raw/sync records: integration

## 4.2 정리 우선순위

모든 테이블을 한 번에 정리하지 말고 우선순위를 준다.

### 우선 1

- 공통 품목 기준
- 사용자/권한 기준
- 생산 계획/실행 기준

### 우선 2

- 품질 첨부/분류 체계
- 재고 스냅샷/실시간 재고 관계

### 우선 3

- 레거시 호환 컬럼 제거
- 보조 통계 구조 정리

## 4.3 마이그레이션 규칙

반드시 사전에 정할 것:

- 기존 데이터 백업 방식
- 롤백 가능 여부
- nullable 허용 기간
- 이행 기간 동안 구버전/신버전 동시 지원 방식
- 운영 데이터 검증 기준

## 5. 프런트 재구축 전에 정의할 것

## 5.1 정보구조(IA)

먼저 결정할 질문:

- 첫 화면은 무엇인가
- 대시보드와 작업 화면을 분리할 것인가
- 모듈별 진입과 역할별 진입 중 무엇을 우선할 것인가
- 현장 사용자와 관리 사용자 경험을 분리할 것인가

## 5.2 공통 UI 규칙

- 테이블 필터 패턴
- 검색 패턴
- 날짜 선택 패턴
- 상세/수정 모달 사용 기준
- 업로드 화면 패턴
- 모바일/태블릿 우선 여부

## 5.3 상태 관리 기준

정의할 것:

- 서버 상태와 UI 상태의 경계
- query key naming 규칙
- form validation 책임
- optimistic update 허용 범위

## 6. AI 연동 전에 정의할 것

AI는 나중 문제처럼 보이지만, 사실 지금 기준을 잘 잡아야 나중에 붙이기 쉽다.

### 먼저 정의할 것

- AI가 접근 가능한 데이터 범위
- AI가 접근하면 안 되는 데이터 범위
- 문서 업로드 허용 범위
- 답변에 근거 링크를 붙일지 여부
- 모델이 직접 생성/수정 가능한지 여부
- read-only 원칙 유지 여부

### 1차 AI 유스케이스 후보

- 일일 생산 요약
- 품질 이슈 요약
- ECO 영향 요약
- 재고 변화 설명
- 문서 검색

## 7. 회의에서 바로 써먹을 질문 목록

아래 질문에 답하면 설계가 많이 선명해진다.

1. 이 시스템의 가장 중요한 사용자는 누구인가
2. 각 사용자가 하루에 가장 자주 하는 작업은 무엇인가
3. 지금 가장 자주 헷갈리는 용어는 무엇인가
4. 같은 품목을 부르는 이름이 부서마다 다른가
5. 생산일 기준은 자정인가 교대인가
6. 반드시 살아 있어야 하는 기존 화면은 무엇인가
7. 없어져도 되는 화면은 무엇인가
8. AI가 처음 도와줘야 하는 업무는 무엇인가
9. AI가 절대 건드리면 안 되는 업무는 무엇인가
10. 3개월 안에 반드시 보여줘야 하는 성과는 무엇인가
