# 사출 이슈 요구사항

## 1. 목적

이 문서는 사출 모니터링에서 생산량 급감, 정지, CT 악화가 발생했을 때
담당자가 원인을 기록하고 추적할 수 있도록 하기 위한 요구사항을 정리한다.

목표는 아래와 같다.

1. 어떤 상황을 이슈 후보로 볼지 정한다.
2. 시스템이 자동 감지하는 것과 사람이 입력하는 것을 나눈다.
3. 사출 담당자가 입력해야 하는 최소 항목을 고정한다.
4. 이후 모델/API/화면 설계의 기준 문서로 쓴다.

## 2. 문제 정의

사출 생산에서는 아래 상황이 반복적으로 중요하다.

1. 시간당 생산량이 갑자기 줄어든다.
2. 생산이 멈춘다.
3. 실제 CT가 기준 CT보다 나빠진다.
4. 왜 그런 일이 일어났는지 모니터링 데이터만으로는 설명되지 않는다.

즉, 수치 데이터만 있어서는 부족하고
"누가, 언제, 왜, 어떻게 조치했는지"를 함께 기록해야 한다.

## 3. 핵심 원칙

1. 생산량 급감/정지는 시스템이 먼저 감지한다.
2. 감지된 후보에 대해 담당자가 사유를 입력한다.
3. 이슈는 계획 작업과 연결되어야 한다.
4. 이슈는 나중에 분석 가능한 코드 체계를 가져야 한다.
5. 자유 텍스트만 두지 말고 유형 분류도 함께 저장한다.

## 4. 이슈 후보 감지 규칙

초기에는 너무 복잡한 이상탐지보다 명확한 규칙 기반으로 시작하는 것이 좋다.

## 4.1 생산량 급감

정의 예시:

- 직전 기준 생산량 대비 현재 slot 생산량이 크게 감소

권장 규칙 예시:

- `actual_vs_target_ratio < 0.7`
- 또는 `delta_qty < baseline_delta_qty * 0.7`

설명:

- baseline은 목표 CT 기준 기대 생산량 또는 직전 이동 평균으로 둘 수 있다.
- 초기에는 목표 CT 기준이 더 단순하다.

## 4.2 생산 정지

구버전 정의 예시:

- 연속 slot 동안 생산량이 0

구버전 규칙 예시:

- `delta_qty = 0` 이 연속 2개 slot 이상

설명:

- 이 절의 예시는 초기 일반 이슈 후보 규칙이다.
- 현재 `/production`, `/mes/monitoring`의 사출 정지/전환 분석은 2분 slot 기준 `10분 이상` 무생산을 사용한다.

## 4.3 CT 악화

정의 예시:

- 추정 CT가 목표 CT 또는 기준 CT보다 악화

권장 규칙 예시:

- `estimated_ct_sec > target_ct_sec * 1.2`

## 4.4 온도/전력 이상

초기에는 참고 신호로만 사용하고, 1차 감지 트리거로는 보조적으로 두는 것이 좋다.

이유:

- 생산량 급감보다 현장 설명력이 낮을 수 있다.
- 설비별 정상 범위가 다를 가능성이 높다.

## 5. 이슈 후보와 실제 이슈의 차이

이 둘을 분리해서 보는 것이 중요하다.

## 5.1 이슈 후보

시스템이 자동으로 판단한 경고 구간

예:

- `10:10 ~ 10:30`
- `3호기`
- `delta_qty 22`
- `baseline 80`
- `estimated_ct 58.2s`

## 5.2 실제 이슈

담당자가 확인하고 등록한 운영 기록

예:

- 금형 문제
- 금형 청소 및 재정렬
- 20분 정지

원칙:

- 모든 후보가 실제 이슈가 되는 것은 아니다.
- 후보는 노이즈가 있을 수 있으므로 사람 확인 절차가 필요하다.

## 9. 입력 화면 요구사항

## 9.1 진입 방식

아래 두 가지 방식이 필요하다.

1. 모니터링 화면에서 특정 slot 경고를 눌러 진입
2. 계획 행 또는 설비 기준으로 수동 등록

## 9.2 자동 채움 항목

모니터링 화면에서 진입하면 아래는 자동 채워야 한다.

- 설비
- 날짜
- 계획 작업
- slot 시작 시각
- 감지 유형
- 감지 수치
- baseline
- 추정 CT

## 9.3 수동 입력 항목

- 이슈 유형
- 상세 설명
- 조치 내용
- 종료 시각
- 결과 상태

## 9.4 화면 동작

권장 동작:

1. 경고 slot 클릭
2. 이슈 등록 모달 열림
3. 자동 수치 표시
4. 담당자 사유 입력
5. 저장 후 모니터링 구간과 연결 표시

## 10. 목록/히스토리 화면 요구사항

사출 이슈는 단건 입력만 있으면 안 되고, 나중에 다시 볼 수 있어야 한다.

필요 조회 기준:

- 날짜
- 설비
- 품목
- 이슈 유형
- 상태
- 등록자

필요 컬럼:

- 시작/종료 시각
- 정지 또는 급감 시간
- 유형
- 상세 사유
- 조치
- 상태

## 11. 분석 요구사항

이슈가 쌓이면 아래 분석이 가능해야 한다.

1. 설비별 가장 많은 이슈 유형
2. 품목별 CT 악화 빈도
3. 자주 멈추는 시간대
4. 평균 복구 시간
5. 계획 미달성과 이슈의 상관관계

초기에는 화면에 바로 다 만들지 않아도 되지만, 데이터 구조는 이 분석을 염두에 두고 잡는 것이 좋다.

## 12. 권한 요구사항

권장 권한:

- `injection.read`
  - 모니터링과 이슈 목록 조회
- `injection.write`
  - 이슈 등록/수정
- `injection.issue.close`
  - 이슈 종료/닫기
- `analysis.read`

초기에는 `injection.write` 하나에 묶어도 되지만, 장기적으로는 닫기 권한을 분리하는 것도 고려할 수 있다.

## 13. 비기능 요구사항

1. 이슈 등록은 빠르게 끝나야 한다.
2. 현장 사용자가 긴 설명을 쓰지 않아도 되도록 유형 선택이 먼저 와야 한다.
3. 모바일 또는 태블릿에서도 입력 가능해야 한다.
4. 자동 감지는 너무 잦은 오탐으로 현장을 방해하면 안 된다.

## 17. 이 문서의 핵심 결론

1. 사출 이슈는 "모니터링 수치 + 사람 설명"의 결합 데이터다.

3. 이슈는 반드시 계획 작업과 연결되어야 나중에 분석 가치가 생긴다.

## 18. 생산계획 완료 기반 정지/전환 판정

2026-05-19 논의 기준으로 사출 정지 구간은 단순한 "무생산"이 아니라 생산계획 진행 상태와 함께 해석한다.

분석 대상 이벤트 타입:

| 타입 | 의미 | 후보/KPI 포함 |
| --- | --- | --- |
| `mold_change` | 금형 교체 | 포함 |
| `core_change` | 코어 교체 | 포함 |
| `tuning` | 사출조건준비 / 调机 | 포함 |
| `production_stop` | 생산 중지 | 포함 |
| `general_stop` | 일반 정지. 이벤트로 생성하지 않고 제외 처리 | 제외 |

원칙:

- 일반 정지는 내부 분류로만 사용하고, 정지/전환 후보 리스트와 KPI에는 노출하지 않는다.
- 후보 리스트와 KPI는 금형 교체, 코어 교체, 사출조건준비, 생산 중지만 대상으로 한다.

초기 판정 규칙:

1. MES 모니터링 형합수에서 10분 이상 생산량이 0인 구간을 정지 후보로 잡는다.
2. 정지 직전의 생산 run이 해당 작업지시 계획 수량 이상을 생산했고 다음 작업지시가 있으면 Part No 판정 규칙에 따라 `금형 교체` 또는 `코어 교체` 후보로 본다.
3. 정지 직전의 생산 run이 해당 작업지시 계획 수량에 미달했더라도 같은 모델/같은 작업지시 생산 도중 멈춘 구간이면 `생산 중지` 후보로 묶고, 사출과가 이슈 사유를 확인한다.
4. 정지 이후 생산이 재개된 시점부터 C/T가 안정적으로 유지되는 첫 구간까지는 `사출조건준비 / 调机` 후보로 본다.
5. 위 판정은 자동 확정이 아니라 `사출과 확인 필요` 상태로 보여주고, 필요시 담당자가 상세 사유와 조치 내용을 입력한다.

Part No 기반 금형/코어 교체 판정:

- 직전 작업지시와 다음 작업지시의 Part No가 모두 11자리이고 비교 가능한 형식이면 Part No를 우선 기준으로 사용한다.
- 11자리 Part No 기준 마지막 두 자리만 다르면 `코어 교체` 후보로 본다.
- 마지막 두 자리 외의 Part No가 달라지거나 Part No 전체가 변경되면 `금형 교체` 후보로 본다.
- Part No가 없거나 11자리 형식이 아니거나 형식이 불명확하면 Part No 차이만으로 판정하지 않고 모델명과 작업지시 변경 여부를 함께 보고 보수적으로 판정한다.
- 보수 판정 결과도 자동 확정하지 않고, 판정 근거와 함께 `사출과 확인 필요` 상태로 남긴다.

2026-05-19 보정 규칙:

- 생산 run 내부에서 계획 수량을 넘겼더라도, 정지 구간이 `금형 교체` 또는 `코어 교체`로 판정되기 전에는 다음 작업지시 실적으로 자동 이월하지 않는다.
- 생산대시보드의 계획별 실적 배분도 같은 기준을 사용한다. 금형/코어 교체 신호가 확인되기 전까지는 계획 수량을 초과한 형합수도 현재 작업지시의 초과 생산으로 배정한다. 예를 들어 A 모델 100개, B 모델 200개 계획에서 120개 생산 후 금형 교체 패턴이 잡히면 A 모델 실적은 120개로 배정하고, B 모델은 교체 후 재가동분부터 집계한다.
- 다음 작업지시의 정지 전 생산량은 교체 후 재가동된 시점부터 새로 누적한다. 예를 들어 1호기 13:22~14:18 교체 후 15:40 정지 전 근거는 14:18 이후 실적만 반영한다.
- 같은 모델/같은 작업지시의 생산 도중 발생한 정지는 작업지시 전환으로 보지 않고 하나의 `생산 중지` 후보로 묶는다.
- `정지 - 조건준비 - 정지 - 조건준비`처럼 짧은 재가동과 정지가 인접 반복되는 패턴은 여러 개의 독립 정지로 쪼개지 않고 하나의 `사출조건준비 / 调机` 후보로 병합한다.
- 병합된 조건준비 이벤트도 자동 확정이 아니라 사출과 확인/메모가 필요한 후보로 남긴다.
- 해당 설비에 당일 생산계획이 없거나, 마지막 작업지시를 완료했고 다음 작업지시가 없는 뒤의 장기 무생산은 `일반 정지`로 본다. 일반 정지는 전환/조건준비 후보 리스트와 KPI에서 제외한다.
- 작업지시 A를 완료한 뒤 다음 작업지시 B가 있더라도 바로 생산하지 않고 작업자가 다른 사출기로 이동했다가 장시간 후 복귀해 B를 생산하는 경우도 `일반 정지`로 본다. 초기 기준은 A 완료 후 B 시작까지 90분 이상 공백이면 전환 후보에서 제외하고, B 시작 시점부터 새 작업지시 실적으로 누적한다.
- 짧은 정지가 반복되는 패턴은 치수/외관 품질검사를 위해 1회 생산 후 검사하는 `사출조건준비 / 调机` 상황이 많으므로, 독립 생산 중지보다 조건준비 병합 후보로 우선 검토한다.
- MES 수집 지연/누락은 UPH를 왜곡하지 않는다면 정지로 보지 않는다. 예를 들어 1분당 1개 생산 중 3분 누락 후 3개가 한 번에 보정 집계되면 정상 운전으로 본다. 초기 구현은 정지 threshold 이상의 0 slot 뒤에 집계된 다음 양수 slot의 산출 속도가 직전 run 평균 속도의 65~160% 범위이면 MES 보정 집계로 보고 run을 끊지 않는다.
- 식사/교대/휴식 시간 정지는 회사 운영 기준상 사용하지 않는다.
- Cavity 오류는 자동 정지 분류에서 다루지 않고 수동 점검 대상으로 둔다.
- 같은 Part No는 Lot가 달라도 원재료/색상이 바뀌지 않는 것으로 보고 별도 전환 후보로 분리하지 않는다.
- 전력/오일온도 변화가 있지만 형합수가 없는 일반 준비/대기 상태는 빈번하므로 일반 정지로 본다.
- 생산 중지의 실제 원인(기계 알람, 금형 이상, 원료 대기, 작업자 부재, 품질 확인 등)은 사출과가 수동 description으로 보완한다.
- 계획 대비 초과 생산이 일정 기준 이상이면 `초과 생산 확인 필요` 플래그를 붙인다. 초기 기준은 계획 대비 105% 이상이고 초과 수량이 5개 이상인 경우다.
- 당일 계획이 없더라도 D+1/D+2 계획이 있는 설비에서 MES 실적이 잡히면 `선행 생산 가능성` 플래그를 붙인다.
- 안정 생산 시작은 초기 기준으로 3개 연속 양수 slot이 있고, 해당 3개 slot의 산출 편차가 평균의 35% 이하일 때로 본다. 이 기준은 `사출조건준비 / 调机`의 종료 시각 추정에만 사용하며 자동 확정은 아니다.

화면 노출 원칙:

- `/production` 생산대시보드는 설비별 정지/전환 후보를 요약해 생산 담당자가 전체 흐름을 보게 한다.
- `/mes/monitoring` MES 모니터링은 선택 설비의 정지/전환 후보를 함께 보여 현장 데이터 확인 중 바로 해석할 수 있게 한다.
- 두 화면은 같은 계산 기준을 사용해야 하며, 화면마다 별도 로직을 만들지 않는다.

초기 시각화:

- KPI: 정지 후보 수, 금형 교체 시간, 코어 교체 시간, 사출조건준비 시간, 생산 중지 시간.
- 설비 카드: 정지/전환 타임라인, 정지 시간대, 판정 근거, 대상 작업지시.
- 판정 근거: 정지 전 누적 생산량, 작업지시 계획 수량, 10분 이상 무생산 여부, Part No 비교 결과, 모델명/작업지시 비교 결과, 안정 생산 시작 추정 시각.

주의:

- 현재 MES 형합수만으로 실제 품목 전환 시각을 완전히 확정할 수는 없다.
- 계획 수량을 넘긴 뒤에도 같은 품목을 계속 찍은 초과 생산인지, 이미 다음 품목으로 넘어간 것인지는 사출과 확인 기록으로 보강해야 한다.
- 장기적으로는 금형/코어 교체 기록, 초품 검사 시각, 표준 C/T를 함께 저장해야 판정 정확도가 올라간다.

검증 체크리스트:

1. 코드 기준은 `frontend-next/src/domains/production/injection-transition-analysis.ts` 한 곳에서 관리한다.
2. `/production`과 `/mes/monitoring`은 같은 `buildInjectionTransitionAnalysis` 결과를 사용한다.
3. 두 화면 모두 D+1/D+2 생산계획을 함께 조회해 `선행 생산 가능성` 플래그를 판단한다.
4. E2E mock은 다음 케이스를 포함해야 한다.
   - 같은 작업지시 중간 10분 이상 정지: `생산 중지`
   - 계획 완료 후 다른 Part No 전환: `금형 교체`
   - 계획 수량 초과 후 금형/코어 교체 신호 발생: 교체 전 초과분은 이전 작업지시 실적으로 배정
   - 11자리 Part No에서 마지막 두 자리만 변경: `코어 교체`
   - 10분 이상 0 slot 뒤 보정 집계되어 UPH가 유지되는 경우: 정지 후보 제외
   - 일반 정지: 후보 리스트와 KPI 미노출
   - 계획 대비 105% 이상 및 5개 이상 초과: `초과 생산 확인 필요`
   - 당일 계획 없음 + D+1/D+2 계획 있음 + MES 실적 있음: `선행 생산 가능성`
5. 기본 검증 명령은 다음 순서로 실행한다.
   - `cd frontend-next && npm run build`
   - Render 백엔드 프록시를 바라보는 프론트 서버에서 `npm run test:e2e`
   - `git diff --check`
6. 검증 후 임시 프론트 서버는 종료하고, 로컬 백엔드 `8000`이 떠 있지 않은지 확인한다.
