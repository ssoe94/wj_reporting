# WJ Reporting 재구축 전략

## 1. 목적

현재 `wj_reporting`는 단순 리포팅 사이트가 아니라 운영 시스템에 가깝다.

- React/Vite 프런트엔드
- Django/DRF 백엔드
- PostgreSQL 기반 데이터 저장
- JWT 인증
- 사출/가공/품질/재고/생산/ECO/관리 기능 운영
- MES 연동 및 스케줄 작업 존재

따라서 이번 재구축의 목표는 "한 번에 새로 만들기"가 아니라 아래 두 가지를 동시에 만족하는 것이다.

1. 사용자에게는 더 깔끔한 UI/UX와 정리된 정보구조 제공
2. 개발자 입장에서는 백엔드와 데이터 구조를 점진적으로 정리

## 2. 현재 상태 요약

현재 코드에서 확인되는 특징은 다음과 같다.

### 강점

- 기능 범위가 이미 넓다.
- 도메인 데이터가 축적되어 있다.
- 생산/MES/재고/품질 흐름이 어느 정도 연결되어 있다.
- 공통 품목 마스터로 발전할 수 있는 `UnifiedPartSpec`가 이미 있다.

### 리스크

- 프런트 라우트와 백엔드 도메인 경계가 명확하게 분리되지 않은 부분이 있다.
- 레거시 호환 필드가 남아 있다.
- 권한 구조는 존재하지만 라우트 접근 로직과 세밀하게 일치한다고 보기 어렵다.
- 시간대 기준이 혼재되어 있다.
- 입력 API, 조회 API, 집계 API가 장기 기준 없이 함께 확장된 흔적이 있다.
- 같은 도메인 개념이 여러 앱에 흩어져 있다.

## 3. 권장 전략

## 전략 한 줄 요약

`기존 운영 시스템은 유지하면서, 새 프런트를 별도 구축하고, 백엔드와 DB는 호환성을 유지한 채 단계적으로 정리한다.`

### 하지 말아야 할 방식

- 기존 사이트를 바로 종료하고 새 시스템으로 한 번에 교체
- DB 스키마를 먼저 크게 갈아엎고 나중에 프런트를 맞추는 방식
- AI 기능까지 한 번에 묶어서 재개발

이 방식은 초기에는 깔끔해 보여도, 실제로는 운영 리스크와 일정 리스크가 매우 크다.

### 추천 방식

1. 새 프런트엔드를 별도 프로젝트 또는 별도 앱 경계로 구축
2. 당분간 기존 Django API를 그대로 사용
3. 새 프런트 요구에 맞춰 API를 정리하되, 기존 화면과 호환성 유지
4. DB는 새 기준 테이블/컬럼/뷰를 추가하는 방식으로 점진 이행
5. AI 서비스는 최종적으로 정리된 read API 또는 read replica 성격의 데이터 계약을 사용

## 4. 시스템 역할 분리

재구축 이후 역할은 아래처럼 나누는 것이 좋다.

### 기존/주 운영 백엔드

- 사용자 입력
- 운영 데이터 저장
- 인증/권한
- MES/재고/배치 연동
- 공통 도메인 규칙

### 새 프런트엔드

- 화면 구조 재설계
- 사용자별 업무 진입 동선 개선
- 모듈 간 탐색 구조 정리
- 입력 화면/조회 화면/대시보드 화면 분리

### 향후 AI 서비스

- 자연어 질의응답
- 생산/품질/재고 요약
- 문서 검색 및 RAG
- 보고서 초안 생성

주의할 점은 AI 서비스가 운영 DB의 직접 쓰기 권한을 가지면 안 된다는 것이다.

## 5. 재구축 범위 정의

이번 재구축의 1차 범위는 아래로 제한하는 것이 좋다.

### 포함

- 프런트 정보구조 재설계
- 핵심 화면 재구축
- API 계약 재정의
- 공통 도메인 기준 정리
- DB 정리 우선순위 정의

### 제외

- 모든 기존 테이블 즉시 재설계
- 모든 화면 동시 교체
- AI 기능 본격 도입
- 외부 연동 전체 재개편

## 6. 우선 정리할 핵심 축

### 6.1 품목 기준 통일

현재 시스템 전반에서 `part_no`가 핵심 키처럼 쓰인다.
재구축에서는 아래를 먼저 확정해야 한다.

- canonical key가 `part_no`인지 `material_code`인지
- 모델명(`model`, `model_code`, `material_name`)의 관계
- 사출/가공/품질/재고에서 같은 품목을 어떻게 매핑하는지
- `UnifiedPartSpec`를 공통 마스터로 승격할지 여부

### 6.2 권한 기준 통일

현재는 모듈 권한과 레거시 권한이 혼재되어 있다.
재구축에서는 아래를 고정해야 한다.

- 메뉴 접근 권한
- 조회 권한
- 편집 권한
- 승인 권한
- 관리자 권한
- 현장 단말 전용 권한

권한은 프런트 임의 판단이 아니라 서버 기준 응답으로 정리하는 것이 좋다.

### 6.3 날짜/시간 기준 통일

현재 시스템에는 UTC 저장, 상하이 기준 계산, 08:00 기준 business date 같은 운영 규칙이 함께 존재한다.
이 부분은 AI 연동 이전에 반드시 정리해야 한다.

- DB 저장 기준: UTC
- 업무 표시 기준: 사업장 현지 시간
- business date 기준: 예를 들어 오전 08:00 교대 기준
- API 입력/출력 포맷: ISO 8601

### 6.4 API 역할 분리

API는 다음 3종으로 나눠야 한다.

1. 입력 API
   - 생성/수정/삭제
2. 조회 API
   - 단건/목록/필터
3. 집계 API
   - 대시보드/분석/요약

현재처럼 하나의 도메인 안에 성격이 다른 엔드포인트가 계속 섞이면 다시 복잡해지기 쉽다.

## 7. 단계별 추진안

## Phase 0. 문서화 및 기준 확정

- 화면 목록 정리
- 현재 API 목록 정리
- 도메인 용어집 작성
- 권한표 작성
- 시간대/업무일 기준 확정
- 공통 품목 기준 확정

## Phase 1. 새 프런트 아키텍처 설계

- 라우팅 구조 재설계
- 레이아웃/네비게이션 규칙 정의
- 모듈별 화면 책임 분리
- 공통 테이블/필터/폼 패턴 정의
- API 클라이언트 계층 표준화

## Phase 2. 백엔드 계약 정리

- 기존 엔드포인트 목록 분류
- 유지할 API / 감쌀 API / 폐기 후보 API 구분
- 응답 스키마 표준화
- 권한 응답 구조 표준화
- 집계 API 재정의

## Phase 3. 데이터 정리

- 공통 마스터 선정
- 중복 컬럼과 도메인 불일치 정리
- 마이그레이션 대상 선정
- 읽기 호환 계층 유지

## Phase 4. 새 프런트 점진 오픈

- 분석/대시보드부터 먼저 교체
- 그 다음 조회 화면
- 마지막으로 입력 화면
- 운영 검증 후 기존 화면 비활성화

## Phase 5. AI 연동

- 새 API 계약 기준으로 read-only AI API 제공
- 문서 RAG 연결
- 요약/질의응답 추가

## 8. 초기 1차 개발 우선순위 추천

처음부터 모든 모듈을 건드리지 말고 아래 순서를 추천한다.

1. 인증/권한 기준 정리
2. 공통 레이아웃과 네비게이션 재설계
3. 분석/대시보드 페이지 재구축
4. 생산계획/실행 조회 화면 정리
5. 품목 마스터 기준 정리
6. 입력 화면 재구축
7. AI 연결용 read API 정리

이 순서가 좋은 이유는 읽기 화면과 대시보드가 구조를 가장 잘 드러내고, 운영 리스크도 상대적으로 낮기 때문이다.

## 9. 성공 기준

재구축이 잘 됐는지는 아래 기준으로 판단한다.

- 새 프런트에서 핵심 사용자 업무가 더 적은 클릭으로 가능하다.
- 같은 도메인 데이터가 서로 다른 이름으로 중복 설명되지 않는다.
- API 계약이 문서 기준으로 예측 가능하다.
- 권한 판단이 프런트 임시 로직이 아니라 서버 계약으로 일관된다.
- 신규 화면 추가 시 기존보다 훨씬 덜 헷갈린다.
- 이후 AI 서비스가 별도 시스템으로 안정적으로 붙을 수 있다.
