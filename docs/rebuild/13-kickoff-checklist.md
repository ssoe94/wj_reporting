# 착수 체크리스트

## 1. 목적

이 문서는 재구축을 실제로 시작하기 전에 확인해야 할 항목을 짧고 실무적으로 정리한 체크리스트다.

쓰는 방법:

- 시작 전에 한 번 훑는다.
- 스프린트 킥오프 때 체크한다.
- 막히는 항목은 별도 결정 문서로 빼서 기록한다.

## 2. 프로젝트 경계

- [ ] 기존 사이트는 운영 유지로 간다.
- [ ] 새 프런트는 별도 구조에서 시작한다.
- [ ] 기존 백엔드는 당분간 공용으로 사용한다.
- [ ] 새 프런트는 우선 기존 API를 읽는 방향으로 간다.
- [ ] AI 서비스는 현재 스코프에서 직접 구현하지 않고 read layer 준비까지만 본다.

## 3. 화면 범위

- [ ] P1 화면 목록이 확정되었다.
- [ ] P1에서 제외할 화면 목록이 정해졌다.
- [ ] `/injection`, `/assembly`, `/quality`는 분해 후 구현한다는 원칙에 합의했다.
- [ ] 조회 중심 화면을 먼저 이관한다는 순서에 합의했다.

권장 P1:

- `/analysis`
- `/production`
- `/production/stats`
- `/sales/inventory`
- `/sales/daily-report`

## 4. 라우트와 모듈

- [ ] 새 프런트 최상위 모듈 구조를 확정했다.
- [ ] `app`, `auth`, `shared`의 책임이 정해졌다.
- [ ] 도메인 모듈 경계가 정해졌다.
- [ ] 현장 단말 흐름은 일반 데스크톱 흐름과 분리한다.

## 5. API 계약

- [ ] 공통 응답 형식 원칙을 정했다.
- [ ] 날짜/시간 필드 표현 규칙을 정했다.
- [ ] 직접 `fetch('/api/...')` 사용을 제거하기로 했다.
- [ ] 품질 화면의 mismatch API 정리 대상이 식별되었다.
- [ ] 생산 계획 API 계약이 다시 확인되었다.

## 6. 권한

- [ ] 최소 capability 목록 초안이 있다.
- [ ] 메뉴 노출 규칙과 API 권한 규칙을 분리해서 적었다.
- [ ] `staff`, `app_admin`, `field_terminal` 차이를 설명할 수 있다.
- [ ] 권한 없는 화면은 숨기고, 서버도 동일하게 막는다는 원칙에 합의했다.

## 7. 데이터 기준

- [ ] `business_date` 정의가 있다.
- [ ] timezone 기준이 있다.
- [ ] `part_no`, `model_code`, `material_code` 정의가 있다.
- [ ] 원천 데이터와 스냅샷/집계 데이터를 구분할 수 있다.

## 8. 개발 방식

- [ ] 한 번에 전체 교체하지 않는다.
- [ ] 화면 단위로 점진 이관한다.
- [ ] 새 프런트에서 공통 컴포넌트와 API 계층을 먼저 만든다.
- [ ] 재사용은 그대로 복사보다 구조에 맞게 이식하는 방식으로 한다.

## 9. 이번 주 첫 작업

- [ ] 새 프런트 작업 위치를 결정한다.
- [ ] 기본 폴더 구조를 만든다.
- [ ] 라우터와 auth 가드를 만든다.
- [ ] 공통 API client를 만든다.
- [ ] `/analysis` 또는 `/production` 한 화면을 새 구조로 옮겨본다.

## 10. 보류 기준

아래 중 하나라도 해당하면 세부 구현 전에 다시 멈춰서 정리하는 편이 좋다.

- [ ] 같은 화면이 두 개 이상의 도메인 책임을 동시에 가진다.
- [ ] 프런트가 호출하는 API 경로가 확실하지 않다.
- [ ] 화면 권한과 서버 권한이 다르게 해석된다.
- [ ] 날짜 기준이 사람마다 다르게 설명된다.
- [ ] 입력 폼이 어떤 마스터 데이터를 기준으로 하는지 불분명하다.

## 11. 시작 선언

아래 문장을 팀의 시작 기준으로 삼아도 좋다.

`기존 사이트는 유지하고, 새 프런트는 공통 골격과 조회 화면부터 점진적으로 구축한다.`
