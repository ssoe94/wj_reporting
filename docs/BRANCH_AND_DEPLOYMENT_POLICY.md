# 브랜치 및 배포 운영 규칙

## 1. 목적

운영 시스템을 중단하지 않고 차세대 프런트엔드를 개발하기 위해 `main`과 `beta`의 역할, 배포 절차, 공용 DB 사용 규칙을 고정한다.

## 2. 환경 구분

| 구분 | Git 브랜치 | Render 서비스 | URL | 배포 범위 |
| --- | --- | --- | --- | --- |
| 운영 | `main` | `wj_reporting`, `wj_reporting_backend` | `https://wj-reporting.onrender.com` | 운영 프런트엔드와 공용 백엔드 |
| 베타 | `beta` | `frontend-next-zjk1` | `https://frontend-next-zjk1.onrender.com` | `frontend-next`만 배포 |

두 프런트엔드는 같은 운영 백엔드와 DB를 사용한다. 베타 브랜치에서는 별도 백엔드나 DB 마이그레이션을 배포하지 않는다.

## 3. 브랜치 역할

- `main`: 현재 운영에 배포 가능한 코드만 유지한다.
- `beta`: 차세대 프런트엔드의 통합 검증 브랜치다. 항상 최신 `main`을 포함해야 한다.
- `shared/<작업명>`: 운영에도 조기 반영할 수 있는 기능이다. `main`에서 만들고 베타 검증 후 동일 브랜치를 `beta`와 `main`에 병합한다.
- `beta/<작업명>`: 차세대 화면 전용 기능이다. `beta`에서 만들고 `beta`에만 병합한다.
- `hotfix/<작업명>`: 운영 긴급 수정이다. `main`에서 만들고 운영 반영 직후 `main`을 `beta`에 병합한다.

`main`과 `beta`에 직접 기능을 개발하지 않는다. 예외는 배포 기반 설정을 최초 정리하는 경우와 승인된 긴급 복구뿐이다.

## 4. 일상 개발 절차

### 베타 전용 기능

1. 최신 원격 상태를 가져온다.
2. `beta`를 기준으로 `beta/<작업명>` 브랜치를 만든다.
3. 구현과 로컬 검증을 완료한다.
4. PR 대상 브랜치를 `beta`로 지정한다.
5. Actions의 프런트엔드 빌드, 린트, 백엔드 호환성 검사를 통과한다.
6. 병합하면 `beta` Actions가 `frontend-next`만 베타 Render 서비스에 배포한다.
7. `build-info.json`과 베타 스모크 테스트가 해당 커밋과 `beta` 브랜치를 확인해야 배포가 성공한다.

### 운영에도 먼저 반영할 기능

1. `main`에서 `shared/<작업명>` 브랜치를 만든다.
2. 동일 기능 브랜치를 먼저 `beta`에 병합해 실데이터로 검증한다.
3. 검증 완료 후 같은 브랜치를 `main`에 병합한다.
4. 같은 커밋을 공유하기 위해 이 경로에서는 임의 cherry-pick이나 squash 복제를 피한다.
5. `main` 배포 후 `main`을 다시 `beta`에 병합해 두 브랜치의 기반을 맞춘다.

### 운영 긴급 수정

1. `main`에서 `hotfix/<작업명>` 브랜치를 만든다.
2. 운영 검증 후 `main`에 병합한다.
3. 운영 배포가 정상인지 확인한다.
4. 같은 날 `main`을 `beta`에 병합한다. 이 단계를 생략하면 최종 전환 시 충돌 위험이 커진다.

## 5. 자동 및 수동 배포

- `main` push: 전체 검증 후 운영 백엔드와 운영 프런트엔드를 배포한다.
- `beta` push: 전체 호환성 검증 후 베타 `frontend-next`만 배포한다.
- PR: 검증만 수행하고 배포하지 않는다.
- 수동 실행: Actions의 `Run workflow`에서 브랜치를 선택하고 `action=deploy`를 선택한다.
- 수동 검증만 필요하면 `action=test-only`를 선택한다.
- `main` 이외 브랜치에서 운영 백엔드 배포 훅을 호출하지 않는다.

필수 GitHub Actions secrets:

- `RENDER_DEPLOY_HOOK_BACKEND`: `main` 운영 백엔드
- `RENDER_DEPLOY_HOOK_FRONTEND`: `main` 운영 프런트엔드
- `RENDER_DEPLOY_HOOK_FRONTEND_NEXT`: `beta` 베타 프런트엔드

## 6. 공용 DB 및 API 규칙

- 베타 화면에서 등록, 수정, 삭제한 데이터는 즉시 운영 데이터가 된다.
- 삭제, 초기화, 일괄 변경에는 확인 절차와 권한 검사를 둔다.
- DB 마이그레이션은 `main`의 운영 백엔드 배포에서만 실행한다.
- 새 API와 필드는 기존 운영 화면이 계속 동작하도록 추가형·하위 호환 방식으로 먼저 배포한다.
- 베타 프런트엔드는 운영 백엔드에 아직 없는 API를 필수 경로로 가정하지 않는다. 필요한 경우 fallback을 제공한다.
- 기존 필드 제거와 비호환 API 변경은 최종 전환과 안정화가 끝난 뒤 별도 작업으로 진행한다.

## 7. 배포 전 체크리스트

- 대상 브랜치가 `main`인지 `beta`인지 확인한다.
- 변경 파일에 `.env`, 토큰, 비밀번호가 없는지 확인한다.
- `frontend` 린트가 오류 없이 끝나는지 확인한다.
- `frontend-next`와 운영 프런트엔드 빌드가 성공하는지 확인한다.
- 백엔드 테스트와 마이그레이션 파일 충돌을 확인한다.
- 공용 DB 쓰기 동작이 운영 사용자에게 미칠 영향을 확인한다.
- Render 서비스 브랜치가 운영은 `main`, 베타는 `beta`인지 확인한다.

## 8. 배포 후 체크리스트

- Actions의 배포와 검증 작업이 모두 녹색인지 확인한다.
- `/build-info.json`의 `commit`과 `branch`가 Actions 대상과 일치하는지 확인한다.
- 운영은 로그인, `/production`, `/mes/monitoring`, API health를 확인한다.
- 베타는 `/production`, `/mes/monitoring`, 공용 API proxy를 확인한다.
- UI가 비어 있거나 이전 버전이면 브라우저 캐시보다 먼저 `build-info.json`을 확인한다.
- 오류 발생 시 새 배포를 반복하기 전에 실패한 job과 Render 로그의 최초 오류를 기록한다.

## 9. 최종 전환 절차

1. 운영 기능 변경을 잠시 동결한다.
2. 최신 `main`을 `beta`에 병합한다.
3. 베타에서 핵심 업무 흐름과 공용 DB 쓰기 동작을 최종 검수한다.
4. 전환 직전 운영 커밋에 릴리스 태그를 생성한다.
5. `beta`에서 `main`으로 PR을 열어 병합한다. `main`을 force push로 덮어쓰지 않는다.
6. 운영 Render 배포와 커밋 검증을 확인한다.
7. 안정화 기간 뒤 구형 화면과 호환 코드를 단계적으로 제거한다.

## 10. 장애 예방 원칙

- 테스트 명령 뒤에 `|| echo`를 붙여 실패를 성공으로 바꾸지 않는다.
- 린트와 백엔드 테스트는 오류가 있으면 배포를 차단한다.
- Render 자동 배포는 끄고 Actions 검증을 통과한 배포 훅만 사용한다.
- 배포 성공 여부는 HTTP 200만으로 판단하지 않고 커밋 SHA와 브랜치까지 확인한다.
- 운영 수정은 반드시 `beta`로 역병합한다.
- 베타 브랜치에서 운영 백엔드와 DB 마이그레이션을 배포하지 않는다.
